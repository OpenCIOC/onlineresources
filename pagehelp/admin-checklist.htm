General Information about Checklist Editing

It is important to carefully read the information contained on the "Edit Checklist / Drop-Down Values" page you are currently working with.

Depending upon the specific list or drop-down type you have chosen to edit, you will see different values on this page. Most versions of the "Edit Checklist / Drop-Down Values" page have a similar format and similar field names, and this help file covers the major variations you will see. If you are working in a multi-member database, please read the section on working with shared and local values to help you understand how to manage your checklists effectively in co-operation with the other CIOC Members in your database.

Note: Changes made to the values on these pages will usually also update the values in existing records; exceptions include Street Type, Postal Box Type, Vacancy Service Title, and Contact Honorific. Changes made to lists that are exceptions will do not alter existing records.

Working in a Multi-Member Database: Shared and Local Values

A multi-member database is one where multiple CIOC Members have their own separate licenses, but share some common setup items and have the ability to share records. If you work in a multi-member database, the setup for your Checklist and Drop-down values becomes a more complicated process, because you may share some or all of these values with other members in your database.

Fully-Shared Checklists

  • A Fully-Shared Checklist is one in which the contents cannot be customized for each member – all members in the same database must use the same values. Some of these values are not currently customizable by the user (CIOC-supplied values), and so all members currently use the same values in their separate databases.
  • Values that are editable (not CIOC-supplied) can only be edited by a Global Super User. Local Super Users may view a list of fully-shared checklist values, but not modify them. Local Super Users may use the link provided to send a notice to the Global Super User(s) requesting a change or new value.
  • Includes: Street Direction, Street Type, Postal Box Type, Activity Status, Billing Address Type, Honorific, Contact Phone Type, Mapping Categories, General Areas of Interest.

Partially-Shared Checklists

  • A Partially-Shared Checklist is one in which each member has some control over the values they wish to use. Some values may be shared by all members, some may be used by only a portion of the members, and some may be local to a single member.
  • Only a Global Super User may create or modify shared values that are available to all users of the database. A Local Super User may elect not to use any shared value by hiding that value for their membership.
  • Local Super Users may add local values that do not already exist. The Local Super user has exclusive control of this local value; it cannot be used or changed by other members. If another member wants to use a local value created by another member, a Global Super User can convert it to be a shared value. Local Super Users must use the link provided to send a notice to the Global Super User(s) to request the value be made shared.
  • Checklist values not in use by the current member will still appear on record details, update forms, etc if the record being accessed already uses those values. The unused values will not be available to that member for adding to records that do not already use the value.
  • Includes: Accessibility, Accreditation, Areas Served, Bus Routes, Certification, Distribution, Extra Checklist A, Extra Checklist B, Extra Checklist C, Extra Checklist D, Extra Checklist E, Extra Checklist F, Extra Drop-down A, Extra Drop-down B, Extra Drop-down C, Extra Drop-down D, Extra Drop-down E, Extra Drop-down F, Extra Drop-down G, Extra Drop-down H, Fees, Fiscal Year End, Funding, Languages, Map Record, Membership Types, Payment Method, Payment Terms, Preferred Currency, Record Quality, Record Type, Schools, Service Level, Type of Care, Type of Program, Vacancy Service Title, Vacancy Target Population, Vacancy Unit Type, Ward

Modifying Values

Text Input Boxes: Names, Codes and Other Values

Each type of checklist has one or more components that make up each value. Some values that can be edited contain only one text input box, generally for the name (i.e. Accessibility) whereas other values contain two or more input boxes for each value being edited (i.e. Distribution contains both a short form Distribution Code as well as an associated longer Distribution name). Most inputs have places to enter name translations if working in a multi-lingual database.

Note that in some cases where an input for a Code or Number is given, it is optional and may not even be used for display. The following Checklists require a Code or Number to be entered:

  • Bus Routes
  • Currency
  • Record Quality
  • Record Type
  • Service Level
  • Ward

Working with Languages

For most checklists, you may specify multiple language values per checklist item. For others, you must pick a single language per checklist item.

For multi-language checklists, the software will normally default to the regular working languages of your database (generally, English and French). If additional languages are available for editing, you will see checkboxes at the top of the form; select the additional language(s) you wish to edit by checking the box next to the language, and they will be added to the form.

Important note: Most checklists require that you specify a name in each language in order for that value to appear when the record is displayed in that language. Exceptions are as follows:

  • Bus Routes - Route number used
  • Languages - First available name used if not available in the current language
  • Preferred Currency - Code used
  • Record Quality - Code used
  • Record Type - Code used
  • Schools - First available name used if not available in the current language
  • Service Level - Code used
  • Vacancy Unit Type - First available name used if not available in the current language
  • Ward - Number used

If you need to specify a name in order to use the checklist item in the language, the box will highlight red if it is left blank.

Order

For many types of Checklists and Drop-downs there is an "Order" column in the edit values page. The display order will determine the order in which values appear in the select list as well as the record itself. Display order numbers may be duplicated; whenever two entries have the same display order number, they will be sorted in alphabetical order of by the Name or Code provided.

Add, Update and Delete

The entire Checklist form (all values) are added, updated, or deleted in a single submission. There is one "Update" button that must be used to accept all changes. Remember: you must always submit all your changes via the Update button, whether you are adding, updating, or deleting.

Add: To create a new entry, click the "Add New Item" button to insert a row for entering new data. You may enter multiple rows per submission.

Delete: To delete an existing entry, check the box in the "Delete" column. If a checkbox is unavailable, it is because the item is in use and cannot currently be deleted.

Street Type, Postal Box Type, and Contact Honorific Checklists

These checklist items work somewhat differently from other types of Drop-downs, in that they are not language-specific and only act as suggested values for records. It is possible to have records in your database that use values other than those in the list - though you generally can't select a new value that is not part of the list - and changes to the list do not impact existing records. In addition, the values in these lists are available ti be used in any language of record, even when they might imply a particular language; this is because the address or contact may do more to suggest the appropriate value than the language used to describe the record.

Searching and Reporting on Checklist Values

Search and Usage

Most checklists have a search icon . Holding your mouse over this icon will display the current usage of this value in the database, including usage by deleted or expired records. Clicking the icon will initiate a search of records using this value in the current View (including deleted / expired records if they are available in the View). If you see the search icon for some values but not others, it means that the values without the search icon are not being used by any records.

Reports

Most checklists will have a link to a print version, which for most lists will include all values as well as date modified / modified by information and usage counts.

Age Groups Editing

The Edit Age Groups page is an administrative tool for creating customized named age ranges that can be used from the Basic Search to identify records serving that age range.

Enabling the end-user to search on age ranges is a three step process:

  1. Populate the Eligibility field of each record in the database with applicable minimum and maximum ages.
  2. Create named age groups through this Edit Age Groups page
  3. Activate the Age Range Search option in the View in which you wish it to appear.

The table on this page contains the following columns:

  • Age Group Name: e.g. Infants, Children, Youth, Adults, Seniors
  • Min Age: The minimum age of this age group. Note that leaving this column blank implies the minimum age is 0. Decimals are allowed.
  • Max Age: The maximum age for this age group. The age specified here is included in the range. Again, if left blank, there is no maximum age for this age group. Decimals are allowed.
  • Chid Care: indicates whether the age group should be used for the specialized child care search page.

Important Note: The age component of the Eligibility field for each record in the database has to be populated with accurate minimum and maximum data, or else search results based on age ranges will be inaccurate.
